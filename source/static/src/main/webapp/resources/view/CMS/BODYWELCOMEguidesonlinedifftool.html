An online diff tool sits between a code editor and a full version-control system in the workflow ladder. The editor reads one version at a time, the version-control system reads the change history of many files, and the diff tool answers the narrower question "what changed between these two specific texts" without either piece of context. Reach for it when you have the before and the after on the clipboard but neither one is in a repository yet - drafting a release note against an earlier draft, comparing a vendor-supplied config file against the production version on disk, or sanity-checking a copy-edit pass against the prior revision before sending the new version back.

The pattern that fails the diff tool is comparing inputs that were not the same shape to begin with - a YAML config against the JSON output it produces, a minified bundle against its unminified source, or a UTF-8 text against the same text saved with different line endings. The diff still runs but the line-by-line view fills with noise that does not correspond to the change the reader cared about. Round-trip the two inputs through the same normaliser first (a JSON parser, a beautifier, or a line-ending converter), then diff the normalised versions; the diff then reads only the changes that matter.
